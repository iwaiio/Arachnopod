# Архитектура данных и обмена

## 1. Назначение документа
Настоящий документ задает формальное описание слоя данных и обмена данными в проекте Arachnopod.

Документ определяет:
- источник описания параметров, команд и алгоритмов обработки;
- разделение на модель системы и системное ПО;
- структуру хранения данных в `MSG_*` массивах и буферах;
- интерфейсы `IMSGGET`, `IMSGSET`, `ISIMGET`, `ISIMSET`;
- правила преобразования `physical <-> wire`;
- протокол обмена по шине;
- жизненный цикл параметров и команд;
- различия между `CS` и не `CS` системами;
- текущее устройство симуляционной шины.

Документ не описывает:
- высокоуровневую миссионную логику аппарата;
- пользовательский интерфейс;
- окончательную board-реализацию транспортного канала.

## 2. Базовые определения

### 2.1. Physical value
Physical value - физическое значение параметра или команды в предметной области.

Примеры:
- температура;
- ток;
- напряжение;
- состояние питания;
- уставка по току или напряжению.

### 2.2. Wire value
Wire value - кодовое представление значения, пригодное для размещения в `MSG_*` массивах и передачи по шине.

Примеры:
- дискретный бит;
- 8-битное целое значение;
- 16-битное целое значение;
- квантизованный код вещественного параметра.

### 2.3. Параметр
Параметр - именованная величина состояния системы.

Параметр может:
- храниться в модели системы;
- дублироваться в системном ПО в `MSG_PRM`;
- передаваться по шине;
- использоваться алгоритмами управления.

### 2.4. Команда
Команда - именованное управляющее воздействие, передаваемое от `CS` к целевой системе.

### 2.5. Локальный параметр
Локальный параметр - внутренняя переменная системы, используемая алгоритмами системного ПО или модели, но не передаваемая по шине.

Локальные параметры:
- могут использоваться через `ISIMGET` и `ISIMSET`;
- не входят в `MSG_PRM`;
- не хранятся в памяти `CS` как часть межсистемного обмена.
- имеют отдельное пространство идентификаторов, предназначенное только для `ISIMGET` и `ISIMSET`;
- задаются в Excel на листе `Local_Params`, а их символьные идентификаторы начинаются с `L`.

### 2.6. Блок данных
Блок данных - элемент полезной нагрузки сообщения размером 16 бит.

Один блок данных может интерпретироваться как:
- одно 16-битное значение;
- два 8-битных значения;
- 16 дискретных битов;
- одно 8-битное значение и 8 дискретных битов.

## 3. Разделение системы на уровни

### 3.1. Системное ПО
Системное ПО - программный слой, который:
- хранит данные в `MSG_*` массивах и буферах;
- выполняет обмен по шине;
- читает и записывает данные через `IMSG*` интерфейсы;
- читает и записывает данные модели через `ISIM*` интерфейсы;
- реализует внутреннюю алгоритмическую логику системы.

### 3.2. Модель системы
Модель системы - слой физических данных и внутренней динамики системы.

Модель:
- хранит параметры в physical-представлении;
- не должна знать о структуре кадров и протоколе шины;
- не работает с `MSG_PRM`, `MSG_CMD`, `MSG_PRM_BUF`, `MSG_CMD_BUF`.

### 3.3. Граница между системным ПО и моделью
Граница проводится следующим образом:
- модель оперирует только physical values;
- системное ПО оперирует physical values и wire values;
- преобразование `physical <-> wire` выполняется только в системном ПО;
- доступ к модели выполняется только через `ISIMGET` и `ISIMSET`.

## 4. Источник описания данных

### 4.1. Единый источник истины
Единым источником описания данных является файл `software/data_base/BD.xlsx`.

В `BD.xlsx` задаются:
- состав систем;
- перечень параметров;
- перечень команд;
- перечень локальных параметров;
- глобальные идентификаторы;
- wire-типы;
- physical-типы;
- алгоритмы обработки данных;
- раскладка по блокам сообщений;
- коэффициенты преобразования и квантования;
- начальные значения model-переменных через столбец `val`.

Глобальные идентификаторы из `Params` и `Commands` используются для wire-метаданных и `IMSG*`.
Локальные идентификаторы из `Local_Params` не входят в wire-метаданные и используются только в слое `ISIM*`.

### 4.2. Локальные параметры и начальные значения в Excel
На основе `BD.xlsx` в модели системы формируются:
- обычные параметры;
- локальные параметры.

Для этого используются листы:
- `Params` - обычные параметры;
- `Commands` - команды;
- `Local_Params` - локальные параметры;
- `Algorithms` - алгоритмы обработки.

Столбец `val` задает начальное значение model-переменной при инициализации.
Это правило одинаково распространяется как на обычные параметры модели, так и на локальные параметры.

## 5. Генерация артефактов из Excel

### 5.1. Генератор
Преобразование `BD.xlsx` в исходные файлы выполняет `software/scripts/bd_parser.py`.

### 5.2. Генерируемые таблицы системного ПО
Генератор формирует:
- `param_tab.hpp` - описание параметров;
- `command_tab.hpp` - описание команд;
- `alg_tab.hpp` - описание алгоритмов обработки;
- `param_comm_list.hpp` - перечень глобальных идентификаторов;
- `local_param_list.hpp` - перечень идентификаторов локальных параметров для `ISIMGET` и `ISIMSET`;
- `sys_list.hpp` - перечень систем;
- `type_list.hpp` - перечень wire-типов;
- `real_type_list.hpp` - перечень physical-типов;
- `alg_list.hpp` - перечень алгоритмов обработки.

`local_param_list.hpp` не является частью wire-метаданных и не используется слоем `IMSG*`.

### 5.3. Генерируемые базовые файлы моделей
Генератор формирует базовые заголовки моделей:
- `sim_cs_base.hpp`;
- `sim_pss_base.hpp`;
- `sim_tcs_base.hpp`;
- `sim_tms_base.hpp`;
- `sim_mns_base.hpp`;
- `sim_ls_base.hpp`.

Эти заголовки содержат:
- обычные параметры модели;
- локальные параметры модели;
- initial values параметров на основе столбца `val`;
- таблицу `PARAMS` для обычных model-параметров;
- таблицу `LOCAL_PARAMS` для локальных model-параметров;
- таблицу `ISIM_PARAMS` как объединенный реестр `PARAMS + LOCAL_PARAMS` для слоя `ISIM*`;
- функцию `init_defaults()`.

### 5.4. Не генерируемый общий слой модели
Файл `sim_base.hpp` не генерируется, так как содержит общую программную логику и инфраструктурные функции, а не описание данных.

## 6. Формальная модель данных

### 6.1. Пространства идентификаторов
В проекте используются два независимых пространства идентификаторов.

#### Глобальное пространство идентификаторов
Глобальные идентификаторы задаются на листах `Params` и `Commands`.

Они используются в:
- generated wire-метаданных;
- `IMSGGET` и `IMSGSET`;
- алгоритмах исполнения команд;
- логике выбора обмена в `CS`.

#### Локальное пространство идентификаторов
Локальные идентификаторы задаются на листе `Local_Params`.

Их свойства:
- символьные имена начинаются с `L`;
- они не входят в `param_tab.hpp`, `command_tab.hpp`, `param_comm_list.hpp`;
- они используются только через `ISIMGET` и `ISIMSET`;
- в generated code они размещаются в отдельном диапазоне `LOCAL_PARAM_ID_BASE`.

Следствие:
- прикладной код не должен смешивать глобальные wire-ID и локальные model-ID;
- ручное дублирование раскладки данных в логике систем недопустимо.

### 6.2. Двухуровневая типизация
В проекте используются два независимых уровня типизации: wire type и physical type.

#### Wire type
Wire type определяет размер и форму размещения значения в сообщении:
- `D` - 1 бит;
- `A` - 8 бит;
- `AP` - 16 бит.

#### Physical type
Physical type определяет физическую семантику значения:
- `bool`;
- `int8_t`;
- `uint8_t`;
- `int16_t`;
- `uint16_t`;
- `float32`;
- `ufloat32`.

### 6.3. Разделение wire и physical
Разделение `wire` и `physical` обязательно, поскольку:
- одно и то же physical value может кодироваться по-разному;
- wire-ширина не определяет физический смысл значения;
- транспортное представление и физический смысл являются разными уровнями описания.

## 7. Алгоритмы обработки данных

### 7.1. Назначение
Алгоритм обработки данных определяет преобразование:
- `physical -> wire` при записи в `MSG_*` массив;
- `wire -> physical` при чтении из `MSG_*` массива.

### 7.2. Поддерживаемые алгоритмы
На текущем этапе используются:
- `NONE`;
- `AN`;
- `AC`;
- `ABC`;
- `AAC`.

### 7.3. Формальная схема кодирования и декодирования
Для дальнейшего описания используются обозначения:
- `x` - physical value;
- `q` - wire value после кодирования;
- `A`, `B`, `C` - коэффициенты алгоритма обработки;
- `q_min`, `q_max` - допустимые границы wire-поля.

Общая схема имеет вид:
1. выполняется преобразование `x -> q` согласно алгоритму;
2. выполняется насыщение `q` в диапазон `[q_min, q_max]`;
3. при чтении выполняется обратное преобразование `q -> x`.

### 7.4. Передача вещественных величин
`float32` и `ufloat32` не передаются по шине в формате IEEE 754.

Передача выполняется в два шага:
1. physical value преобразуется в целочисленный код `q`;
2. код `q` записывается в wire-поле шириной 8 или 16 бит.

Следовательно, передача вещественных значений является передачей дискретизированного кода.

### 7.5. Смысл алгоритмов обработки
На уровне документа алгоритмы обработки интерпретируются следующим образом:
- `NONE` - значение передается без дополнительного преобразования;
- `AN` - используется коэффициент масштабирования;
- `AC` - используется смещение;
- `ABC` - используется масштабирование и смещение;
- `AAC` - используется альтернативная линейная форма преобразования.

Точная формула определяется таблицей алгоритмов, генерируемой из `BD.xlsx`.

### 7.6. Насыщение диапазона
Если результат преобразования `physical -> wire` выходит за допустимый диапазон wire-поля, применяется насыщение:
- при отрицательном переполнении используется минимально допустимое значение;
- при положительном переполнении используется максимально допустимое значение.

Отказ от записи не используется.

### 7.7. Погрешность квантования
Квантование порождает погрешность восстановления physical value после обратного преобразования.

Генератор:
- валидирует корректность алгоритмов;
- вычисляет и печатает оценку погрешности там, где она может быть определена аналитически.

## 8. Интерфейсы доступа к данным

### 8.1. Функциональная матрица интерфейсов

| Интерфейс | Источник | Приемник | Преобразование | Работа с локальными данными |
| --- | --- | --- | --- | --- |
| `IMSGGET` | `MSG_*` | системное ПО | `wire -> physical` | нет |
| `IMSGSET` | системное ПО | `MSG_*` | `physical -> wire` | нет |
| `ISIMGET` | модель | системное ПО | отсутствует | да |
| `ISIMSET` | системное ПО | модель | отсутствует | да |
| `ICMDEXEC` | `MSG_CMD` и системное ПО | модель | зависит от алгоритма команды | да |

### 8.2. IMSGGET
`IMSGGET` парсит данные из `MSG_*` массивов и использует преобразование `wire -> physical`.

Функции `IMSGGET`:
- поиск метаданных по ID;
- определение положения значения в блоке данных;
- извлечение wire value;
- преобразование `wire -> physical`.

`IMSGGET` одинаково применяется как к параметрам, так и к командам.

### 8.3. IMSGSET
`IMSGSET` записывает данные в `MSG_*` массивы и использует преобразование `physical -> wire`.

Функции `IMSGSET`:
- поиск метаданных по ID;
- определение положения значения в блоке данных;
- преобразование `physical -> wire`;
- запись wire value в `MSG_PRM` или `MSG_CMD`.

### 8.4. ISIMGET
`ISIMGET` парсит данные из модели системы в системное ПО без преобразований.

Свойства `ISIMGET`:
- работает только с моделью;
- не использует преобразование `physical <-> wire`;
- может работать как с обычными, так и с локальными данными.

### 8.5. ISIMSET
`ISIMSET` записывает данные из системного ПО в модель системы без преобразований.

Свойства `ISIMSET`:
- работает только с моделью;
- не использует преобразование `physical <-> wire`;
- может работать как с обычными, так и с локальными данными.

### 8.6. ICMDEXEC
`ICMDEXEC` - слой типизированного исполнения команд.

Назначение слоя:
- вынести повторяющиеся шаблоны исполнения команд;
- сократить объем повторяющегося кода в системах;
- стандартизовать типовые алгоритмы выполнения команд.

Индивидуальная логика конкретных команд остается системно-специфичной.

## 9. Структуры хранения данных в системном ПО

### 9.1. Общий состав структур
Для всех систем системное ПО содержит:
- `MSG_PRM`;
- `MSG_CMD`;
- `MSG_PRM_BUF`;
- `MSG_CMD_BUF`;
- `MSG_HEADER`.

### 9.2. Матрица назначения структур

| Структура | Не `CS` система | `CS` |
| --- | --- | --- |
| `MSG_PRM` | исходящие параметры системы | агрегированный прием параметров от подчиненных систем |
| `MSG_CMD` | принятые команды для системы | исходящие команды для подчиненных систем |
| `MSG_PRM_BUF` | буфер отправки параметров | буфер приема параметров |
| `MSG_CMD_BUF` | буфер приема команд | буфер отправки команд |
| `MSG_HEADER` | текущий заголовок кадра | текущий заголовок кадра |

### 9.3. Матрица владения данными

| Слой | Обычные параметры | Команды | Локальные параметры | Представление |
| --- | --- | --- | --- | --- |
| Модель не `CS` системы | да | косвенно через состояние | да | physical |
| Системное ПО не `CS` системы | да | да | да через `ISIM*` | physical + wire |
| `MSG_PRM` не `CS` системы | да, кроме локальных | нет | нет | wire |
| `MSG_CMD` не `CS` системы | нет | да | нет | wire |
| Модель `CS` | локальные данные `CS` | нет как отдельный транспортный слой | да | physical |
| Системное ПО `CS` | да, как принятые данные | да, как формируемые воздействия | да через `ISIM*` | physical + wire |
| `MSG_PRM` в `CS` | да, как агрегированные данные подчиненных систем | нет | нет | wire |
| `MSG_CMD` в `CS` | нет | да | нет | wire |

### 9.4. MSG_HEADER
`MSG_HEADER` - структурированное представление заголовка кадра.

Текущий формат заголовка:

| Поле | Ширина, бит | Назначение |
| --- | --- | --- |
| `parity` | 1 | контроль четности заголовка |
| `addr` | 5 | адрес назначения |
| `flag` | 2 | тип сообщения |
| `marker` | 3 | служебное поле протокола |
| `nblocks` | 5 | число 16-битных блоков полезной нагрузки |

### 9.5. Значения поля `flag`

| Значение | Семантика |
| --- | --- |
| `0` | `CMD_REQ` |
| `1` | `CMD_RESP` |
| `2` | `DATA_REQ` |
| `3` | `DATA_RESP` |

### 9.6. Константы протокола
Для текущей симуляционной реализации используются следующие протокольные константы:

| Константа | Значение | Назначение |
| --- | --- | --- |
| `SOF` | `0xAA` | признак начала кадра |
| `EOF` | `0x55` | признак завершения кадра |
| `ARP_BUS_MAX_DATA_BLOCKS` | `31` | максимальное число 16-битных блоков в `DATA` |

### 9.7. Размеры буферов в `CS`
Размеры рабочих массивов и буферов в `CS` не совпадают.

Для команд:
- `MSG_CMD` содержит полное командное пространство `CS` по всем подчиненным системам;
- `MSG_CMD_BUF` имеет размер массива команд той не `CS` системы, у которой этот массив максимален.

Для параметров:
- `MSG_PRM` содержит полное параметрическое пространство `CS`, разбитое по диапазонам систем;
- `MSG_PRM_BUF` имеет размер массива параметров той не `CS` системы, у которой этот массив максимален.

## 10. Буферная дисциплина и commit

### 10.1. Общий принцип
Данные, принятые с шины, не должны записываться непосредственно в рабочие массивы `MSG_PRM` и `MSG_CMD`.

Корректная последовательность:
1. данные принимаются в `MSG_*_BUF`;
2. структура кадра и завершение кадра проверяются;
3. только после корректного `EOF` выполняется commit в `MSG_*`.

### 10.2. Формальные инварианты
Для слоя данных и обмена принимаются следующие инварианты:
- модель не работает с `MSG_*` структурами;
- локальные параметры не передаются по шине;
- `IMSGGET` и `IMSGSET` не работают с локальными параметрами;
- `ISIMGET` и `ISIMSET` могут работать с локальными параметрами;
- неадресованная система не записывает полезную нагрузку в `MSG_PRM_BUF` и `MSG_CMD_BUF`;
- commit в рабочий массив возможен только после корректного `EOF`;
- `CS` отправляет только команды и принимает только параметры;
- не `CS` системы принимают только команды и отправляют только параметры.

### 10.3. Причины использования буферов
Буферизация необходима для защиты от:
- неполного кадра;
- некорректного завершения кадра;
- частично принятых данных;
- изменения рабочих массивов во время активной передачи.

## 11. Протокол обмена по шине

### 11.1. Структура кадра
Кадр состоит из четырех частей:
- `SOF`;
- `HEADER`;
- `DATA`;
- `EOF`.

### 11.2. Транзакционные сценарии
Используются два сценария:
- `CMD_REQ -> CMD_RESP`;
- `DATA_REQ -> DATA_RESP`.

### 11.3. Формальная последовательность байтов в кадре
Последовательность байтов имеет вид:
1. `SOF`;
2. два байта `HEADER`;
3. `2 * N_BLOCKS` байтов полезной нагрузки;
4. `EOF`.

Следовательно, полная длина кадра в байтах равна:
`L = 1 + 2 + 2 * N_BLOCKS + 1`.

### 11.4. Таблица фаз кадра

| Фаза | Передающая сторона | Принимающая сторона | Изменяемые структуры |
| --- | --- | --- | --- |
| `SOF` | выдает признак начала кадра | ожидает старт кадра | шина |
| `HEADER` | передает заголовок | считывает адрес, тип и `N_BLOCKS` | `MSG_HEADER` |
| `DATA` | передает полезную нагрузку | пишет в буфер или пропускает payload | `MSG_PRM_BUF` или `MSG_CMD_BUF` |
| `EOF` | завершает кадр | подтверждает корректность завершения | commit или отказ от commit |

### 11.5. Поведение неадресованной системы
Если кадр не адресован системе, система обязана:
- принять и разобрать `HEADER`;
- определить `N_BLOCKS`;
- пропустить все байты полезной нагрузки без записи в `MSG_PRM_BUF` и `MSG_CMD_BUF`;
- дождаться `EOF`.

Это необходимо для сохранения синхронизации по структуре кадра.

### 11.6. Примеры раскладки блока данных

| Конфигурация блока | Интерпретация |
| --- | --- |
| `AP` | одно 16-битное значение |
| `A + A` | два 8-битных значения |
| `A + 8D` | одно 8-битное значение и 8 дискретов |
| `16D` | 16 дискретных значений |

### 11.7. Временная диаграмма `CMD_REQ -> CMD_RESP`

```text
Такт T:     CS записывает следующий байт кадра команды в шину
Такт T+1:   целевая система читает этот байт
...
После EOF:  данные из MSG_CMD_BUF коммитятся в MSG_CMD
Далее:      блок выполнения CMD обрабатывает команду
Затем:      система формирует и передает CMD_RESP
```

### 11.8. Временная диаграмма `DATA_REQ -> DATA_RESP`

```text
Такт T:     CS передает байт запроса DATA_REQ
Такт T+1:   целевая система читает байт запроса
...
После EOF:  система подготавливает MSG_PRM_BUF
Далее:      система передает DATA_RESP
После EOF:  CS коммитит MSG_PRM_BUF в диапазон MSG_PRM соответствующей системы
```

## 12. Ошибки обмена и восстановление

### 12.1. Типовые ошибки
В слое обмена существенны следующие ошибки:
- `timeout`;
- `invalid_eof`;
- частично принятый кадр;
- некорректный заголовок;
- неуспешная передача команд;
- неуспешная передача параметров.

### 12.2. Таблица ошибок и реакций

| Ошибка | Условие возникновения | Реакция |
| --- | --- | --- |
| `timeout` | ответный кадр не был получен в допустимое число тактов | commit не выполняется, инициатор фиксирует ошибку |
| `invalid_eof` | на фазе завершения кадра получен байт, отличный от `EOF` | commit не выполняется, кадр отвергается |
| частично принятый кадр | обмен прерван до завершения `DATA` или `EOF` | рабочие `MSG_*` не изменяются |
| некорректный заголовок | заголовок не может быть корректно интерпретирован | кадр не должен приводить к commit |
| неуспешная передача команд | команда не подтверждена корректным ответом | `CS` сохраняет или обновляет `MSG_CMD_BUF` по специальному правилу |
| неуспешная передача параметров | параметры не были подтверждены корректным обменом | не `CS` система заново обновляет `MSG_PRM_BUF` из `MSG_PRM` |

### 12.3. Общие правила реакции
При некорректном обмене:
- commit в рабочие `MSG_*` массивы не выполняется;
- рабочие массивы сохраняют предыдущее корректное состояние;
- логика повторной передачи определяется ролью системы.

### 12.4. Не `CS` система при ошибке передачи параметров
Если передача параметров завершилась некорректно, `MSG_PRM_BUF` повторно обновляется из `MSG_PRM` и используется для следующей попытки передачи.

### 12.5. `CS` при ошибке передачи команд
Если передача команд завершилась некорректно, `MSG_CMD_BUF` обновляется из `MSG_CMD` по правилу:
- если команда в `MSG_CMD != 0`, она перезаписывает соответствующее значение в `MSG_CMD_BUF`;
- если команда в `MSG_CMD == 0`, прежнее значение в `MSG_CMD_BUF` сохраняется.

## 13. Не `CS` система: общая структура

### 13.1. Общая декомпозиция
Любая не `CS` система делится на два уровня:
- системное ПО;
- модель системы.

### 13.2. Декомпозиция системного ПО
Системное ПО не `CS` системы делится на две функциональные части:
- блок хранения wire-данных и обмена по шине;
- блок внутренней логики системы.

### 13.3. Декомпозиция модели
Модель не `CS` системы делится на два блока:
- блок данных, хранящий параметры системы, включая локальные;
- алгоритмический блок модели, выполняющий локальные вычисления на основе параметров.

## 14. Не `CS` система: блок внутренней логики

### 14.1. Состав
Блок внутренней логики системы включает:
- системный алгоритм;
- блок выполнения `CMD`;
- блок обновления `MSG_PRM`.

### 14.2. Системный алгоритм
Системный алгоритм выполняет алгоритмические действия системы независимо от внешних команд `CS`.

Системный алгоритм использует:
- `ISIMGET` для чтения параметров модели, включая локальные;
- `ISIMSET` для записи параметров модели, включая локальные.

### 14.3. Блок выполнения `CMD`
Блок выполнения команд отвечает за обработку команд, находящихся в `MSG_CMD`.

Алгоритм работы:
1. перебираются все системные команды по идентификаторам;
2. каждая команда читается из `MSG_CMD` через `IMSGGET`;
3. если команда равна `0`, она пропускается;
4. если команда не равна `0`, выполняется алгоритм команды;
5. при успешном выполнении команда затирается в `0` через `IMSGSET`.

Алгоритм команды прописывается вручную и может:
- изменять параметры модели через `ISIMSET`;
- использовать другие параметры модели, включая локальные, через `ISIMGET`.

### 14.4. Блок обновления `MSG_PRM`
Блок обновления `MSG_PRM` отвечает за перенос не локальных параметров модели в `MSG_PRM`.

Алгоритм работы:
1. перебираются все идентификаторы параметров системы;
2. для каждого не локального параметра значение читается из модели через `ISIMGET`;
3. значение записывается в `MSG_PRM` через `IMSGSET`.

## 15. Не `CS` система: жизненный цикл команд

### 15.1. Общий принцип
В не `CS` системах команды только принимаются с шины.

### 15.2. Прием и commit
Жизненный цикл команды:
1. командные данные принимаются с шины в `MSG_CMD_BUF`;
2. если обмен завершен корректно, выполняется commit из `MSG_CMD_BUF` в `MSG_CMD`;
3. commit выполняется по правилу ненулевой перезаписи:
   - если команда в `MSG_CMD_BUF != 0`, она перезаписывает предыдущее значение в `MSG_CMD`;
   - если команда в `MSG_CMD_BUF == 0`, запись в `MSG_CMD` не выполняется.

### 15.3. Исполнение команды
После commit команда проходит следующий путь:
1. блок выполнения `CMD` считывает команду из `MSG_CMD` через `IMSGGET`;
2. выполняет алгоритм команды;
3. изменяет параметры модели через `ISIMSET`;
4. при необходимости использует дополнительные параметры модели через `ISIMGET`;
5. при успешном завершении затирает команду в `0` через `IMSGSET`.

### 15.4. Полный жизненный цикл команды
Типичный жизненный цикл команды, отправленной из `CS`, имеет вид:
`передача по шине -> запись в MSG_CMD_BUF -> запись в MSG_CMD -> блок выполнения CMD -> запись параметров в модель системы -> алгоритмический блок модели реагирует на новое состояние`.

## 16. Не `CS` система: жизненный цикл параметров

### 16.1. Общий принцип
В не `CS` системах параметры только отправляются на шину.

В `MSG_PRM` записываются все обычные параметры системы. Локальные параметры в `MSG_PRM` не входят.

### 16.2. Формирование параметров
Жизненный цикл параметров:
1. блок данных модели хранит physical values;
2. блок обновления `MSG_PRM` читает их через `ISIMGET`;
3. значения записываются в `MSG_PRM` через `IMSGSET`;
4. при обмене все содержимое `MSG_PRM` полностью копируется в `MSG_PRM_BUF`;
5. передача на шину выполняется из `MSG_PRM_BUF`.

### 16.3. Поведение после обмена
Если обмен завершен корректно, дополнительные действия не требуются.

Если обмен завершен некорректно, `MSG_PRM_BUF` повторно обновляется из `MSG_PRM` и используется для следующей попытки передачи.

### 16.4. Полный жизненный цикл параметров
Типичный жизненный цикл параметров, передаваемых в `CS`, имеет вид:
`блок данных модели -> блок обновления данных системного ПО -> запись в MSG_PRM -> копирование в MSG_PRM_BUF при обмене -> отправление данных по шине`.

## 17. Порядок выполнения системного ПО не `CS` системы

### 17.1. Целевая последовательность
Порядок выполнения блоков системного ПО задается следующим образом:
1. системный алгоритм;
2. блок выполнения `CMD`;
3. блок обновления `MSG_PRM`;
4. обмен.

### 17.2. Смысл порядка
Такая последовательность означает:
- сначала система изменяет свое внутреннее состояние;
- затем обрабатывает внешние управляющие воздействия;
- затем отражает новое состояние модели в `MSG_PRM`;
- затем выполняет обмен по шине уже на актуальных данных.

### 17.3. Формальный смысл этапов

| Этап | Источник данных | Результат |
| --- | --- | --- |
| системный алгоритм | модель | изменение physical-состояния модели |
| блок выполнения `CMD` | `MSG_CMD` и модель | изменение physical-состояния модели, очистка выполненных команд |
| блок обновления `MSG_PRM` | модель | обновление wire-представления параметров в `MSG_PRM` |
| обмен | `MSG_PRM_BUF`, `MSG_CMD_BUF`, шина | прием или передача кадра |

## 18. `CS`: общая архитектура

### 18.1. Назначение
`CS` является управляющей системой и master шины.

### 18.2. Роль `MSG_*` массивов в `CS`
Для `CS` справедливо следующее:
- `CS` отправляет на шину только команды из `MSG_CMD_BUF`;
- `CS` принимает с шины только параметры в `MSG_PRM_BUF`.

## 19. `CS`: принятие параметров

### 19.1. Прием в буфер
При передаче параметров от целевой системы блоки данных записываются в `MSG_PRM_BUF`, начиная с нулевого элемента буфера.

### 19.2. Commit в рабочий массив
Если передача завершена корректно, параметры копируются из `MSG_PRM_BUF` в диапазон блоков внутри `MSG_PRM`, соответствующий системе-источнику.

Следовательно, `MSG_PRM` в `CS` является агрегированным хранилищем параметров подчиненных систем.

### 19.3. Формальный инвариант диапазонов параметров
Для каждой подчиненной системы в `MSG_PRM` существует фиксированный диапазон блоков.

Инвариант:
- данные, принятые от системы `S`, могут быть записаны только в диапазон `MSG_PRM[S]`;
- запись параметров одной системы в диапазон другой системы недопустима.

## 20. `CS`: передача команд

### 20.1. Формирование буфера передачи
Для передачи команд определенный диапазон блоков из `MSG_CMD`, соответствующий целевой системе, копируется в `MSG_CMD_BUF`, начиная с нулевого элемента буфера.

После копирования диапазон этой системы в `MSG_CMD` затирается в `0`.

### 20.2. Корректная передача
Если обмен завершен корректно, дополнительные действия не требуются.

### 20.3. Некорректная передача
Если обмен завершен некорректно, `MSG_CMD_BUF` обновляется из `MSG_CMD` по правилу:
- если команда в `MSG_CMD != 0`, она перезаписывает соответствующее значение в `MSG_CMD_BUF`;
- если команда в `MSG_CMD == 0`, прежнее значение в `MSG_CMD_BUF` сохраняется.

Эта логика необходима для того, чтобы:
- не потерять ранее сформированные команды после неудачного обмена;
- не пропустить новые команды, добавленные в `MSG_CMD` после неудачной передачи.

### 20.4. Конфликтующие команды
Возможна ситуация, при которой в `MSG_CMD_BUF` остается старая команда, а в `MSG_CMD` появляется новая, логически конфликтующая команда.

Пример:
- сначала была сформирована команда включения;
- затем обмен завершился неудачно;
- после этого была сформирована команда выключения.

Разрешение такого конфликта не должно выполняться в транспортном слое. С этой ситуацией должна разбираться сама целевая система в своем блоке выполнения `CMD`. Алгоритм обработки конфликта может быть уникален для каждой системы.

### 20.5. Формальный инвариант диапазонов команд
Для каждой подчиненной системы в `MSG_CMD` существует фиксированный диапазон блоков.

Инвариант:
- в `MSG_CMD_BUF` попадает только диапазон команд выбранной целевой системы;
- команды других систем не должны присутствовать в активном буфере передачи.

## 21. `CS`: структура системного ПО

### 21.1. Общая декомпозиция
Системное ПО `CS` делится на два главных блока:
- блок хранения wire-данных и обмена;
- блок выполнения алгоритма системы.

### 21.2. Блок хранения wire-данных и обмена
Этот блок отвечает за:
- хранение `MSG_CMD`, `MSG_PRM`, `MSG_CMD_BUF`, `MSG_PRM_BUF`;
- формирование и разбор кадров;
- передачу команд;
- прием параметров;
- копирование данных между рабочими массивами и буферами.

### 21.3. Блок выполнения алгоритма системы
Блок выполнения алгоритма системы делится на:
- блок локальных алгоритмов `CS`;
- блок верхнего уровня;
- блок оператора обмена;
- блок ПО управления системами;
- блок Управляющих Воздействий.

### 21.4. Блок локальных алгоритмов `CS`
Блок локальных алгоритмов `CS` предназначен для внутренней логики, связанной только с `CS`. На текущем этапе находится в разработке.

### 21.5. Блок верхнего уровня
Блок верхнего уровня определяет глобальное поведение `CS`, влияет на обмен и на ПО управления системами. На текущем этапе находится в разработке.

### 21.6. Блок оператора обмена
Блок оператора обмена задает порядок обмена с системами в формате списка задач.

Этот блок определяет:
- с какой системой выполнять следующий обмен;
- какой тип транзакции требуется;
- когда допускается переключение на другую задачу обмена.

### 21.7. Блок ПО управления системами
Блок ПО управления системами:
- работает на основе данных, хранящихся в `MSG_PRM`;
- выполняет управление подчиненными системами;
- формирует команды для подчиненных систем;
- записывает команды в `MSG_CMD`.

Главная работа `CS` с данными и `MSG_*` массивами сосредоточена именно в этом блоке.

Генерация команд в `CS` должна происходить только здесь.

### 21.8. Блок Управляющих Воздействий
Блок Управляющих Воздействий принимает команды от оператора и обеспечивает ручное управление аппаратом.

Этот блок не должен напрямую подменять протокол обмена. Его задача - формировать воздействия для системного ПО `CS`.

## 22. `CS`: модель системы

### 22.1. Общий принцип
Модель `CS` организована аналогично моделям остальных систем, но все данные внутри модели `CS` являются локальными.

### 22.2. Декомпозиция модели `CS`
Модель `CS` содержит:
- блок локальных данных;
- блок локальной логики модели.

## 23. Общая главная функция системного ПО
Унифицированная главная функция системного ПО реализуется в `i_sys_algo`.

На концептуальном уровне она объединяет:
- обработку времени;
- выполнение системного алгоритма;
- выполнение `CMD`;
- обновление `MSG_PRM`;
- обмен;
- обработку power-команд;
- логирование.

Конкретный прикладной смысл команд, параметров и алгоритмов остается системно-специфичным.

## 24. Симуляционная шина

### 24.1. Назначение
`mock_bus` используется только для симуляции транспортного канала.

### 24.2. Модель передачи
Симуляционная шина передает не более одного байта за один системный такт.

Для устранения зависимости от порядка выполнения потоков в одном такте в симуляции используются два состояния байта:
- `visible` - байт, доступный для чтения в текущем такте;
- `pending` - байт, записанный в текущем такте и предназначенный для чтения в следующем такте.

На границе такта выполняется перенос `pending -> visible`.

### 24.3. Граница ответственности шины
`mock_bus` не определяет смысл данных.

Смысл данных должен оставаться в:
- `IMSGGET`;
- `IMSGSET`;
- `ISIMGET`;
- `ISIMSET`;
- `ICMDEXEC`;
- `i_sys_algo`.

## 25. Board-транспорт

### 25.1. Инвариантность смыслового уровня
Смысл параметров, команд и идентификаторов должен быть инвариантен относительно физического транспорта.

От выбора транспортного канала не должны зависеть:
- модель данных;
- идентификаторы;
- алгоритмы обработки;
- структура `MSG_PRM` и `MSG_CMD`;
- прикладная семантика `CMD_REQ/CMD_RESP` и `DATA_REQ/DATA_RESP`.

### 25.2. CAN
CAN может рассматриваться как будущий нижний транспортный уровень board-реализации.

CAN не должен переопределять:
- смысл параметров;
- смысл команд;
- правила преобразования `physical <-> wire`.

Его задача - заменить только физический транспортный уровень.

## 26. Реализованные и разрабатываемые компоненты

### 26.1. Реализовано
На текущем этапе реализованы:
- `BD.xlsx` как единый источник описания данных;
- генерация таблиц и базовых файлов моделей;
- инициализация model-переменных из столбца `val`;
- генерация локальных идентификаторов для слоя `ISIM*`;
- типизированные интерфейсы `IMSGGET` и `IMSGSET`;
- типизированные интерфейсы `ISIMGET` и `ISIMSET`;
- поддержка `bool`, целых типов и вещественных параметров через квантование;
- поддержка алгоритмов `NONE`, `AN`, `AC`, `ABC`, `AAC`;
- буферная схема `MSG_*_BUF -> MSG_*`;
- общий алгоритм обмена в `i_sys_algo`;
- рабочая связка `CS <-> PSS` в симуляции.

### 26.2. В разработке
В разработке находятся:
- расширение описания локальных параметров в `BD.xlsx` для остальных систем;
- дальнейшая унификация исполнения команд;
- локальные алгоритмы `CS`;
- верхний уровень логики `CS`;
- доведение остальных систем до полнофункционального обмена;
- окончательная board-реализация транспорта;
- интеграция будущего CAN-уровня.
